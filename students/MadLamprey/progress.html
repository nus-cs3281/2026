Summary

My CS3282 work this semester broke into two loosely sequential explorations, both anchored to the CS2103T iP:

  1. Java + AI integration: a few weeks experimenting with adding AI features directly inside an iP-style Java app using Langchain4j, with local and hosted LLMs.
  2. iP Increment Reviewer: ideation, implementation, and iterative redesign of a GitHub Action that posts AI-generated code reviews when students push increment tags.
Week Achievements
1 (12-18 Jan) - Picked Langchain4j as the in-app AI framework (briefly compared with Spring AI; Langchain4j had the better tutorial coverage at the time).
- Got a basic chatbot working inside Bruno: user types text, Langchain4j hands it to a hosted model, response streams back. The shape of the "AI inside a Java app" wiring was the main thing I wanted to understand.
2 (19-25 Jan) - Replaced the chatbot with a constrained help command: when the user types help <something>, the LLM is responsible for the entire response. Useful as a first test of how much prompt scaffolding it takes to keep a model on topic.
- Started sketching agent mode: instead of the user typing a structured Duke command, the LLM reads the user's natural-language input and decides which Duke command to invoke.
3 (26 Jan-1 Feb) - Got agent mode working end-to-end with Langchain4j's tool-calling flow (8ee688b). Brittle on smaller models, fine on the larger ones.
- Wrapped phase 1.
- Drafted Tutorial on Agentic Development in iP.
4 (2-8 Feb) - Phase 2: can the hosted API be replaced with a local LLM? Started with JLama because it's pure-Java inference, no Python dependency. Got a small model running end-to-end with Bruno's agent mode. Latency was acceptable for short prompts; quality was the limiting factor on anything non-trivial.
- Surveyed Ollama, vLLM, and a couple of others; most have better model coverage than JLama but pull in more deployment complexity.
- Drafted Tutorial on Local LLMs in Java App.
5 (9-15 Feb) - Tested Prof's reference llama.cpp setup. Quality was meaningfully better than JLama at the same parameter count, but the Java integration story is rougher. For an in-app feature shipped to students, neither was attractive enough to justify replacing a hosted API.
- Wrapped phase 2.
6 (16-22 Feb) - Started thinking about phase 3. Ran a small ideation pass: what does the iP feedback loop actually look like, where is it slow, and where could AI plug in usefully? Listed candidate ideas (generate test inputs, score the diff, summarise commits, propose questions for TAs) and ranked them by "would a student notice if this didn't run."
7 (2-8 Mar) - First version of the reviewer. Started with a simple shape: on tag push, compute the diff against the previous tag, send it to the LLM with the increment requirements, post the LLM's reply as a commit comment.
- First implementation of "requirement text" was scraping the iP webpage live at runtime. Worked locally, but the GitHub Actions runner hit the page intermittently and the parse was brittle.
- Ideated that the reviewer must never write code for students and should be Socratic by design. The system prompt got rewritten to enforce questions over corrections.
8 (9-15 Mar) - Ditched live scraping. Switched to a static, pre-loaded requirements.json keyed by increment tag: the same data, but reproducible and editable per cohort.
- Several rounds of system-prompt iteration: the smaller models I was testing on would default to giving direct corrections. Took a few attempts to consistently get observations framed as questions.
9 (16-22 Mar) - Shipped the first reviewer (d1fd74c): workflow YAML, review.py, and the requirements database all landed together. Posting target was a commit comment.
10 (23-29 Mar) - Hardened the deployed reviewer. Replaced the live webpage fetch with the static requirements.json keyed by tag (0b8ae78) so the workflow no longer depended on the iP page being reachable from the runner.
- Removed leftover code paths from earlier prototypes (9abf314).
- Bumped to a more recent Groq model for slightly better question quality (bb3c131).
- Bug fixes (a13ebd6, 3049b7d).
11 (30 Mar-5 Apr) - First revised attempt after Prof's feedback: an iP Companion (ip-companion-proposal.md) that would turn the review issue into a persistent conversation thread aware of the student's full iP history.
12 (6-12 Apr) - Switched the posting target from commit comment to a GitHub Issue per increment, and in the same refactor added support for follow-up comments on that issue (the precondition for Companion-style Q&A) (cd55cb3). Commit comments were easy to miss and didn't surface in the Issues tab where students actually look; issues also gave a natural place for the Q&A thread.
- Two test commits on master (1b468b8, 887189f) to trigger the workflow against my own repo and watch the issue thread end-to-end.
- Wrote up the full enhancement document (increment-reviewer.md) covering forward-looking warnings, recurring-pattern escalation, and the Companion.
13 (13-19 Apr) - Reframed the design constraint. Every mode the reviewer offers must depend on something a chat tab cannot do. Wrote and discarded a few proposal drafts (a behavior verifier, then an adversarial bug-hunter) before landing on a multi-mode design.
- Drafted the proposal for 3-mode system.
- Built a standalone ip-reviewer repo rather than keeping the package in-tree on the iP fork so that instructors can install it into any iP repo via install.sh and customise rubric/standard files per cohort without touching code.
- Live pilot on MadLamprey/ip against a force-tagged A-Jar snapshot
- Bug fixes

Lead & Technical Contributions

  • iP Reviewer (v2): Designed, built, and piloted the redesigned GitHub Action that runs three on-demand reviewers on a student's iP fork: an Adversarial Tester (auto on tag push), a Rubric Mapper (/rubric), and a Code Quality reviewer (/quality). Each mode is built around a capability a generic chat tab cannot replicate (executing the student's code, reading their whole repo at once).
  • iP Reviewer (v1): Designed, built, and shipped the original Socratic reviewer that posts on every increment tag push. Included idempotent re-tagging, switch from commit comment to issue, and follow-up comment handling.
  • Java + AI framework evaluation: Hands-on with Langchain4j across three increasingly autonomous in-app AI features (chatbot -> AI help -> full agent mode).
  • Local-LLM evaluation: Hands-on with JLama, plus a desk survey of Ollama and a couple of others.
  • Tutorials for CS2103T students: Two tutorials on setting up AI in the iP: one for using Langchain4j to build in-app features, and one for evaluating local LLMs as a replacement for hosted APIs.
